Foundational Context and LLM Philosophy

Focus on outcomes, not debates about “real intelligence”

When working with Large Language Models (LLMs), prioritize results and functionality instead of arguing whether they “think” like humans.

Main ideas

  • LLMs generate text token-by-token (word-by-word), which can resemble how humans build thoughts incrementally.
  • Industry value comes from leveraging strengths, not overcomplicating the system with impractical concerns.
Practical mindset Token-by-token generation Engineering focus

Framework Landscape: The LangChain Family

Frameworks to manage LLM interactions (emerged ~2022)

Modern LLM development often relies on frameworks that organize prompts, tools, memory/state, evaluation, and orchestration.

Key components

  • LangChain: Early, popular framework for building LLM apps and streaming prompts/data to models.
  • LangGraph: Graph-based orchestration (nodes, edges, state, transitions). Built on the same execution stack as LangChain.
  • LangSmith: Typically used for tracing, evaluation, debugging, and observability of LLM apps.
  • LangFlow: Visual/low-code style building of LangChain-style pipelines.
Why these frameworks matter
They reduce boilerplate, standardize patterns (chains, tools, memory, retrieval), and make complex LLM workflows easier to build, test, and monitor than raw API calls.
LangChain LangGraph Observability Orchestration

LangChain Core: Chains and Pipelines

A “chain” is a pipeline of linked steps

LangChain’s core abstraction is the chain: a structured pipeline that moves input through multiple stages.

What a chain does

  • Links steps such as: Prompt → LLM → Output parser/formatter.
  • Hides boilerplate: manual API requests, JSON parsing, and output shaping.
  • Runs the pipeline via a single execution method (commonly invoke).
Takeaway: Chains standardize repeatable workflows so you can focus on behavior and quality instead of glue code.
Pipeline Invoke Parsing Boilerplate reduction

Tool Calling: Defining and Registering Tools

Tools are functions with clear inputs, outputs, and descriptions

Tool calling (also known as function calling) lets an LLM request that external code runs a specific function. It is often considered a prerequisite for more advanced orchestration patterns.

Tool basics

  • A tool is a function: it accepts parameters and returns a result.
  • Tools can be simple (math) or complex (calling a weather API with a location parameter).
  • In LangChain, tools are commonly defined using a dedicated tool annotation (for example, @tool).

Why the description is critical

  • Tool definitions are sent to model providers (for example, OpenAI) as structured JSON: name, parameters, and description.
  • The LLM heavily relies on the description to choose the right tool and fill arguments correctly.
  • Frameworks generate the required JSON schema so you don’t craft it manually.
Tool schema Descriptions matter Parameters

Tool Calling Flow: The LLM as a Router

The model decides the call; your framework executes it

The LLM cannot run functions by itself. Instead, it suggests a tool call, and the surrounding framework executes it. The overall interaction is handled as one continuous session/flow.

End-to-end sequence

  1. Request and identification: The user sends a prompt; the model receives tool definitions (requires a tool-calling-capable model).
  2. Routing decision: The model selects a tool (like a service registry) and extracts arguments from the text.
  3. External execution: The framework looks up the real function and runs it with the extracted parameters.
  4. Result feedback: The framework sends the tool result back to the model as a tool message.
  5. Final answer generation: The model uses the original context plus the tool result to produce a natural-language response.
Fallback behavior
If the model cannot confidently select a tool, it replies directly in natural language (similar to a default case in a switch statement).
Routing Two-pass response Session context

When Tool Calling Is Worth It

Use tools to access data/capabilities outside the model

Tool calling is valuable when the model needs information it was not trained on or needs to perform real actions.

Strong use cases

  • Checking internal company policies, state, or private knowledge.
  • Retrieving inventory and stock by storage location.
  • Calling external APIs (weather, finance, maps, etc.).
  • Performing actions (booking a restaurant, updating a ticket, sending a message).

When it is not necessary

  • Basic math is usually not a strong reason; many modern LLMs can do it natively.
Pattern: In complex scenarios, the model may decide it needs a sequence: “get weather” → “use weather result” → “recommend outfit.”
External data Actions Private context

Advanced Context: MCP (Model Context Protocol)

Scaling tool ecosystems beyond a few functions

The Model Context Protocol (MCP) is a deeper topic (often worth its own session). It relates to tool usage, but expands the ecosystem and management model.

Key idea

  • MCP is similar in spirit to tool chaining, but it supports registering many more tools from various sources.
  • Tools are often managed via injected configuration rather than only being defined programmatically in code.
  • Tools can be treated like a pluggable catalog that a model-enabled application can access consistently.
MCP Tool registry Configuration-driven

Tool Chaining: Solving Multi-Step Problems

The model selects and sequences multiple tools

Tool chaining means the model can determine and execute a sequence of tools to solve a multi-step natural-language problem.

What was demonstrated

  • Given a multi-step word problem (addition, multiplication, then summation), the model identified the correct sequence of operations.
  • The tool chain was implicitly defined by the prompt, representing a very high level of abstraction.
  • The framework managed the sequence end-to-end (in this context, LangChain was used).
Model capability note
The demonstration model was GPT-4o mini and successfully executed the tool chain. The point: smaller models can still perform useful chained tool behaviors when the tooling and prompts are structured well.
Multi-step reasoning Tool sequencing High abstraction

Agents and Error Management

Agents replace human work—expect statistical error

Agents are not deterministic software components. They are closer to replacing human labor and therefore can make mistakes.

Agents vs. procedural code

  • Procedural code (traditional programming) is designed to behave the same way every time.
  • Agents can vary and sometimes fail, because they are probabilistic/statistical systems.

Statistical error is inevitable

  • The goal is to minimize error rates, not to eliminate them entirely.
  • Replacing critical deterministic logic (for example, certain SQL operations) with agents can be risky.
  • Higher throughput (24/7 automation) can raise the total number of errors simply because many more actions occur.

Reducing risk

  • Use validation agents (a second agent checks the first agent’s outputs, logs, and decisions).
  • Add guardrails: schema checks, allow-lists, unit tests, and human review for high-impact actions.
Reliability Validation agents Guardrails

Prompt Engineering: Patterns and Descriptions

Better prompts and better tool descriptions drive better outcomes

Prompt engineering matters most when orchestrating tools and agents. Clarity and structure strongly influence output quality.

Most important lever: tool descriptions

  • The tool description is often the single most important input for tool selection and correct argument extraction.
  • Write descriptions like a specification: what it does, when to use it, and constraints.

Optimal prompt patterns

  • Different tasks have different “best” prompt structures.
  • Example pattern for image generation prompts: subject → location → objects → relationships → camera → photographic style.
  • Suboptimal patterns usually produce suboptimal results.

Prompt engineering may be a transitory skill

  • Prompt engineering emerged early in LLM adoption, similar to how SEO emerged with search engines.
  • It may evolve into newer disciplines (for example, context engineering) as systems become more capable and automated.
Prompt patterns Tool descriptions Context engineering

Prompt Injection and Subliminal Signals (Security)

Invisible inputs can manipulate model behavior

LLMs can be manipulated by adversarial inputs—sometimes embedded in ways humans may not notice. This creates real security risk.

Invisible or “subliminal” prompt injection

  • Models can read and act on transparent/invisible text embedded in images, even when a human viewer cannot see it.
  • This resembles “subliminal perception” experiments (hidden messages flashed too quickly for conscious perception).

Why it matters

  • Fraud risk: hidden instructions can attempt to override user intent and exfiltrate sensitive information.
  • Creative use: creators could embed invisible “watermark instructions” to guide how an image should be processed later.
Practical mitigation ideas
Treat images and documents as untrusted input. Apply content sanitization, model-side instruction hierarchy, allow-listed tool use, and robust logging/monitoring for suspicious instruction patterns.
